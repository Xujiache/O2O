## 一、项目规划

### 1.1 项目基本信息

| 项目 | 内容 |
|---|---|
| 项目名称 | O2O 外卖 + 跑腿一体化平台 |
| 需求文档 | `O2O外卖+跑腿一体化平台 需求规格说明书.md`（V1.0，2026-04） |
| 项目性质 | 同城本地生活即时配送 O2O 平台（外卖点餐 + 同城跑腿双业务线） |
| 开发环境 | Mac / Windows + Node.js 20+ + pnpm + Docker |
| 保密等级 | 内部公开 |

### 1.2 端与技术栈（严格对齐 PRD §5 技术栈规范）

| 端 / 模块 | 技术栈 | 部署目标 |
|---|---|---|
| 用户端 | uni-app + Vue3 + Pinia + uView Plus | 微信小程序 |
| 商户端 | uni-app + Vue3 + Pinia + uView Plus | iOS / Android APP |
| 骑手端 | uni-app + Vue3 + Pinia + uView Plus | iOS / Android APP |
| 管理后台 | Vue3 + Vite + Element Plus + Pinia + Vue Router（基于 Art Design Pro 精简版，已就位于 `管理后台/`） | Web |
| 后端服务 | **NestJS + TypeScript**（模块化微服务，PRD 明确允许的高效方案） | Linux 容器 |
| 数据库 | MySQL 8.0+ 主库 / Redis 6.0+ 缓存 / 时序库存轨迹 | 独立集群 |
| 中间件 | RabbitMQ / MinIO / Redisson | 独立集群 |
| 第三方 | 高德地图、微信开放平台 & 支付、支付宝、极光推送、短信 | SaaS |
| 部署运维 | Docker + Kubernetes + Nginx + Jenkins + Prometheus + Grafana | 私有云 |

### 1.3 后端框架选型说明

PRD 允许 SpringCloud/SpringBoot 或 NestJS。本项目选择 **NestJS**：
- 与前端 uni-app / Vue3 **技术栈统一**（TypeScript），团队协作与人员复用效率最高
- 原生 **模块化 + 依赖注入 + 微服务 + OpenAPI**，开发效率显著高于 Java 体系
- 生态（TypeORM、BullMQ、Passport、Cache Manager）契合 PRD 全部技术需求

## 二、实施方案

### 2.1 9 阶段开发路线图

| 阶段 | 任务名 | 目标 | 核心交付物 |
|---|---|---|---|
| P1 | 项目初始化 | 技术栈落地、代码仓库、开发规范 | 各端脚手架、README、CI 骨架 |
| P2 | 数据库设计 | 全业务数据模型 | ER 图、建表脚本、字典、Redis Key 规划 |
| P3 | 后端基础服务 | 统一认证、用户中心、消息、文件、地图 | 5 大基础服务 + OpenAPI 文档 |
| P4 | 后端业务服务 | 订单、商品、商户、派单、支付、财务、营销、评价 | 8 大业务服务 + 闭环联调 |
| P5 | 用户端开发 | 首页/外卖/跑腿/订单/个人中心/支付消息 6 模块 | 微信小程序完整包 |
| P6 | 商户端开发 | 登录/店铺/订单/商品/财务/数据统计 6 模块 | 商户 APP（iOS+Android） |
| P7 | 骑手端开发 | 登录/接单/配送/钱包/考勤/消息 6 模块 | 骑手 APP（iOS+Android） |
| P8 | 管理后台开发 | 10 大管理模块（数据大盘→客服风控） | 管理后台完整功能 |
| P9 | 集成测试部署 | 功能/性能/安全/合规验收，容器化上线 | 上线包、测试报告、运维手册 |

### 2.2 每个阶段的 6A 工作流（对应 7 份文档）

| 文档 | 用途 |
|---|---|
| `ALIGNMENT_[任务名].md` | 需求对齐（回溯 PRD 原文、拆解、边界确认、待澄清问题） |
| `CONSENSUS_[任务名].md` | 达成共识（范围、技术、交付标准、风险） |
| `DESIGN_[任务名].md`    | 详细设计（架构/模块/数据/接口/UI） |
| `TASK_[任务名].md`      | 任务拆解 WBS（优先级、依赖、负责人、工时） |
| `ACCEPTANCE_[任务名].md`| 验收标准（功能/性能/安全/交付物） |
| `FINAL_[任务名].md`     | 阶段总结（交付物清单、验收结果、遗留问题） |
| `TODO_[任务名].md`      | 待办清单（动态跟踪，每完成一项即更新） |

### 2.3 关键里程碑（基于 PRD §6 验收标准）

- **M1**：P1~P2 完成 → 基础设施就绪
- **M2**：P3~P4 完成 → 后端全业务闭环可调通
- **M3**：P5~P8 完成 → 四端功能联通
- **M4**：P9 完成 → 通过 PRD §4 非功能需求与 §6 验收标准，正式上线

### 2.4 硬性约束（用户规则）

1. 绝不允许项目延期 —— 按文档节点执行，风险提前识别，同步更新文档
2. 绝不允许超出计划 —— 范围严格匹配规划，扩容先改规划
3. 绝不允许出错 —— 多轮自检（编译、测试），错误先暂停、定位根因、修复再推进
4. 技术问题优先用 MCP Context7 检索最新文档 / 示例；不决则查官方文档；**严禁编造**

| **用户（你）** | 调度者：发起阶段指令、在 Cascade 与执行 Agent 之间传递提示词和结果 |
| **Cascade（我）** | 规划者 + 审查者：生成阶段开发提示词、审查代码、输出问题清单 |
| **执行 Agent** | 编码者：根据提示词写代码、根据问题清单修 BUG |

### 6.2 三段式循环（**多员工并行开发 + 单员工集中修复**）

```
① 发起       用户 → Cascade："开始写 PX 阶段开发提示词"
                Cascade → 用户：同时输出 N 份提示词（N=2 或 3）
                                《Agent 开发提示词_PX_员工A》
                                《Agent 开发提示词_PX_员工B》
                                《Agent 开发提示词_PX_员工C》（可选）

② 并行执行   用户 → Agent_A / Agent_B / Agent_C：分别粘贴
                3 人各自开发自己负责的子任务，文件域互不重叠
                员工 A 默认为组长，承担集成验证 + 进度文档更新
                Agent_X → 用户：开发完成

③ 整体审查   用户 → Cascade："PX 全部完成，请审查"
                Cascade → 用户：《整合审查报告 + 问题清单_PX_第 N 轮》
                                （把 A/B/C 全部代码合在一起审，不分人）
                
                如 PASS → 进入 P(X+1)
                如 FAIL → 回到 ④

④ 单员工修复 Cascade → 用户：《修复提示词_PX_第 N 轮》（**仅一份**，给一个 Agent）
                用户 → 单个 Agent：粘贴修复提示词
                Agent → 用户：修复完成
                用户 → Cascade："修复完成，请复审"
                回到 ③
```

> 修复阶段不再并行，原因：① 修复多为跨文件的逻辑对齐 / 集成 BUG，并行易再撞冲突；② 单员工承担可清晰追责；③ 修复任务量通常较小，单员工可在合理时间内完成。

### 6.3 用户触发指令（只需说其中一句）

| 指令 | Cascade 动作 |
|---|---|
| `开始 PX 提示词` / `生成 PX 开发提示词` | 按 §6.10 拆分规则输出 N 份（员工 A/B/C）开发提示词 |
| `PX 全部完成，请审查` / `审查 PX` | 整合审查 A/B/C 全部代码 + 输出《整合审查报告_PX_第 N 轮》 |
| `修复完成，请复审` / `复审 PX` | 复审 + 输出结论（PASS 或新一轮问题清单） |
| `PX PASS，进入 P(X+1)` | 更新 §3.1 为 🟢，自动生成下一阶段员工 A/B/C 提示词 |

### 6.7 循环终止条件
- Cascade 审查报告 §B 结论为 **PASS**
- `docs/PX_*/FINAL_*.md` 所有勾选完成、签字
- 用户说 `PX PASS，进入 P(X+1)`
- Cascade 更新 §3.1 状态为 🟢、§3.3 追加日志，并立即生成 P(X+1) 开发提示词

### 6.8 轮次上限与降级
- 同一阶段若审查 ≥ 3 轮仍未 PASS，Cascade 主动输出《根因分析报告》+ 是否需要调整 DESIGN_*.md 的建议
- 若调整 DESIGN，先更新阶段文档、再重发开发提示词（不是继续改代码）

### 6.9 阶段 PASS 后 git 入库（强制）

每当一个阶段（PX）被 Cascade 判定 **PASS** 并将 §3.1 对应行升为 🟢 后，**必须立即**完成一次 git 入库，作为阶段性里程碑归档。

| 项 | 要求 |
|---|---|
| 触发条件 | §3.1 状态由 🟡 / ⬜ 升为 🟢 |
| 执行人 | Cascade 在复审 PASS 同一轮次内引导用户授权后直接执行 |
| 工作流文件 | `.windsurf/workflows/stage-git-commit.md`（详细步骤） |
| 远端 | `origin` = `https://github.com/Xujiache/O2O.git` |
| 分支 | `main`（主开发线；后续如引入 feature 分支再更新此条） |
| commit 规范 | 严格遵循 `commitlint.config.cjs`（Conventional Commits；type ∈ feat/fix/docs/style/refactor/perf/test/build/ci/chore/revert；scope ∈ 用户端/商户端/骑手端/管理后台/后端/部署/docs/workspace） |
| 推送后 | 在 §3.3 追加一条「P(X) 已同步至 GitHub」日志 |

**五步固定顺序**（详见工作流文件）：

1. `git status -s` 核对改动
2. 确认/配置 `origin` remote
3. `git add -A`
4. `git commit -m "feat(<scope>): P(X) [阶段名] PASS — <关键交付摘要>"`
5. 首次 `git push -u origin main`；后续 `git push`

**硬性约束**：
- 禁止 `--force` 到 `main`
- 禁止跨阶段混合 commit（一个阶段一个 commit 或多 commit 但严格围绕该阶段）
- push 失败先排查 credentials / 网络，不得绕过

### 6.10 员工编制（单 Agent 模式，基于 P3 实战调整）

> **V2.0 重大变更（2026-04-19，基于 P3 实战教训）**：从 P4 开始**默认单 Agent 主导**开发与修复。多员工并行模式（V1.6/V1.7）经 P3 实战发现集成成本（跨员工 import 路径、重复键、占位文件、API 兼容性等 9 处协调修复）超过并行收益，且首轮审查易漏检（如 P3-REVIEW-01 没第一时间识别 MessageService stub 占位）。多员工模式保留为"可选模式"，仅适用于纯独立、无 import 耦合的子模块场景（如多端 UI 并行）。

#### 6.10.1 员工编制（默认单 Agent）

| 编制 | 适用阶段 | 默认 |
|---|---|---|
| **单员工** | **本项目所有阶段默认** | ✅ V2.0 起默认 |
| 2 员工（A/B） | 仅当任务严格分前后端互独立 | 可选 |
| 3 员工（A/B/C） | 仅当任务严格分独立子模块（如多端 UI）| 可选 |

#### 6.10.2 拆分原则

| 维度 | 规则 |
|---|---|
| **文件隔离** | 不同员工写不同文件 / 子目录，**严禁**同文件并发改 |
| **领域内聚** | 同一业务域（如 D1 账号、D4 订单）由同一员工完成，避免跨人协作语义模糊 |
| **依赖前置** | 强依赖任务（如建库、初始化）放在员工 A，且明确标注"完成后通知 B/C 启动" |
| **组长责任** | 员工 A 默认组长，额外承担：① 全量集成验证 ② 更新 `说明文档.md §3.3` 进度日志 |
| **工时均衡** | 各员工工时差距控制在 ±20% 以内 |
| **TODO 文件** | 共用 1 份 `TODO_PX_*.md`，每人**只勾选自己负责的行**，组长在最后汇总检查 |
| **必读文档** | 全员都必须读完 PRD + 阶段 6 份文档（不可省略） |
| **同步点** | 提示词中明确标注"同步点"：何时需要等待其他员工完成才能开始下一任务 |

#### 6.10.3 模板 A 的多员工版本（结构）

每份员工提示词在 §6.4 模板 A 基础上追加 3 段：

1. **§0.5 你的角色**：员工 X / 是否组长 / 你的产出范围 / 你的同伴是谁
2. **§3 任务清单**：仅列你负责的 WBS 编号 + 标题（不列其他员工的）
3. **§3.5 与其他员工的同步点**：明确"等待 X 完成 Y 才能开始 Z"或"完成 W 后通知 X 启动"

#### 6.10.4 修复阶段（单员工）

修复阶段**不并行**：

- Cascade 整合 A/B/C 全部代码进行审查，输出**一份**整合问题清单
- 修复提示词只产出**一份**，由用户挑选一名 Agent 执行（推荐用组长 A，因其熟悉集成全貌）
- 修复完成后再次整合审查，循环至 PASS

#### 6.10.5 入库（PASS 后）

不论开发时几名员工，PASS 后入库仍按 §6.9 单次 commit 完成（commit message 中可在 body 注明 "by A/B/C"）。

#### 6.10.6 完全并行模式（推荐，本项目默认）

为了最大化开发速度，本项目默认采用**完全并行**：三个员工自第 0 秒起即可同时动手，不设串行等待。前提与执行规则：

| 项 | 规则 |
|---|---|
| 自验证环境 | 每个员工**各自起一个独立 docker MySQL 容器**自验证，端口/库名/容器名不冲突 |
| 建库 helper | 自验证时可临时使用最小化 `CREATE DATABASE` 语句，**集成时**统一改用组长产出的 `00_init.sql` |
| ER 图等"看似有依赖"的任务 | 一律基于 `DESIGN_*.md` 画/写，**不依赖**他人 SQL 已落地（DESIGN 是权威基准，SQL 只是其落地） |
| TODO 勾选 | 三人共用 `TODO_PX_*.md`，**只勾自己负责的行**；如发生 git 冲突，组长 A 在合并时统一处理 |
| 集成验收（唯一串行点） | 全员完成后由组长 A 在干净 docker MySQL 内按文件名顺序执行 00→10 + seeds 01→04 + timescale，要求 0 报错；这是**唯一**串行任务 |
| 互相 review | 鼓励三人在自己完成后翻看其他两人的 SQL（同步给用户即可），发现疑似冲突立即报告 |

> 完全并行后，原 §6.10.2 中"强依赖任务放员工 A 并通知 B/C"这条仅在「无法各自起 docker」的退化场景生效；正常场景下不适用。
