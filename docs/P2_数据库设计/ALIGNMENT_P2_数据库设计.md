> 本文件用于 P2 阶段**需求对齐**，严格基于《O2O外卖+跑腿一体化平台 需求规格说明书 V1.0》，不做任何偏离。

## 一、PRD 原文回溯

### 1.1 数据相关技术栈（PRD §5）

| 类型 | 要求 |
|---|---|
| 主数据库 | MySQL 8.0+ |
| 缓存数据库 | Redis 6.0+ |
| 时序数据库 | 存储骑手定位轨迹（PRD 原文明确要求） |
| 对象存储 | MinIO / 阿里云 OSS（非关系，但需 URL 存库） |

### 1.2 对数据能力的业务要求

- **高并发订单**：≥ 1000 TPS（PRD §4.1.2）
- **大数据量查询**：支持百万级订单数据快速检索（PRD §4.1.5）
- **骑手定位**：≥ 10 秒/次上报（PRD §4.1.4），PRD §3.5.1.5 要求"骑手实时定位上报与存储"
- **订单状态推送**：延迟 ≤ 3s（PRD §4.1.4）
- **数据安全**：敏感信息加密存储（PRD §4.3.1）
- **备份**：每日全量 + 增量，异地存储（PRD §4.4.3）

## 二、P2 任务范围

### 2.1 in-scope

1. 全量表结构设计（MySQL 建表 DDL）
2. 索引策略（主键、唯一键、二级索引、覆盖索引）
3. 分库分表策略（订单、骑手轨迹、日志类大表）
4. Redis Key 命名规范与典型场景
5. 时序数据库方案（骑手轨迹）
6. 字典数据初始化（订单状态、违规类型、行业分类等）
7. 数据迁移与备份策略
8. 敏感字段加密字段清单（PRD §4.3.1）
9. ER 图（分领域）
10. 初始化 SQL 脚本与 seed 数据

### 2.2 out-of-scope

- TypeORM Entity 代码实现（留到 P3）
- 数据库性能压测（留到 P9）
- 具体备份工具部署（留到 P9）

## 三、关键约束对齐

| 约束 | 数据库设计响应 |
|---|---|
| 高并发 1000+ TPS | 订单表按月水平分表 + Redis 扣减 + RabbitMQ 削峰 |
| 百万级查询 ≤ 1s | 按时间、用户、商户、骑手维度建复合索引；热点查询读 Redis |
| 订单状态推送 ≤ 3s | 状态变更写 MySQL → MQ 广播 → 各端 WebSocket/订阅消息 |
| 骑手定位 10s/次 | 时序库存轨迹；Redis 存最新位置（带 TTL） |
| 敏感信息加密 | 手机号/身份证/银行卡采用 AES-GCM + 字段级脱敏 |
| 数据无丢失 | InnoDB + binlog + 每日全量 + 每小时增量 |
| 故障恢复 ≤ 30min | 主从 + 自动切换 + 读写分离 |

## 四、待澄清问题（默认建议）

| 编号 | 问题 | 默认建议 |
|---|---|---|
| Q2.1 | 时序库选 InfluxDB 还是 TimescaleDB？ | **TimescaleDB**（PostgreSQL 插件，SQL 兼容性好） |
| Q2.2 | 外卖+跑腿订单合一张表还是分表？ | **分表**：`order_takeout` + `order_errand` + `order_master`（公共字段基表），用 `order_no` 全局唯一 |
| Q2.3 | 订单分表策略？ | **按月分表 + 订单号含分片键**，保留 24 个月热数据在 MySQL，历史归档到 OSS + Hive |
| Q2.4 | 是否使用 ShardingSphere？ | 暂不引入，应用层路由，后期业务上规模再评估 |
| Q2.5 | UUID 还是雪花 ID？ | **雪花 ID**（Long，自增趋势，索引友好），订单号人类可读（时间戳+分片+序号） |
| Q2.6 | 敏感字段加密：字段级 or 全表加密？ | **字段级 AES-GCM**（按需解密，影响小）+ 身份证/银行卡密文 + 明文前缀脱敏 |
| Q2.7 | 软删除标记？ | 统一 `is_deleted`（0/1） + `deleted_at`（可空） |
| Q2.8 | 时间字段？ | 统一 `created_at` / `updated_at` / `deleted_at`（`DATETIME(3)`），应用层写入 |
