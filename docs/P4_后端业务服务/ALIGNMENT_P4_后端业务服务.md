# ALIGNMENT_P4_后端业务服务

> 本文件用于 P4 阶段**需求对齐**，严格基于《O2O外卖+跑腿一体化平台 需求规格说明书 V1.0》§3.5.2。

## 一、PRD 原文回溯（§3.5.2 业务核心服务层）

| 编号 | 服务 | PRD 原文能力 |
|---|---|---|
| S1 | 订单服务 | 外卖/跑腿订单创建、状态流转、库存扣减、取消/退款逻辑、订单生命周期全流程管控 |
| S2 | 商品服务 | 商品分类、商品信息、规格、库存、上下架管理 |
| S3 | 商户服务 | 商户入驻、店铺管理、营业状态、评价管理、营销工具管理 |
| S4 | 配送调度服务 | **核心智能派单算法**、订单分发、顺路单匹配、骑手运力调度、配送轨迹管理、异常订单流转 |
| S5 | 支付服务 | 对接微信支付/支付宝；下单支付、退款、对账、分账、资金冻结/解冻、支付状态同步 |
| S6 | 财务结算服务 | 分账规则执行、商户/骑手账单生成、提现管理、平台营收统计、对账管理 |
| S7 | 营销服务 | 优惠券、满减、折扣、活动规则配置与执行、发放与核销管理 |
| S8 | 评价与售后服务 | 用户评价管理、售后工单、纠纷仲裁、投诉处理服务 |

## 二、跨引用业务流程（PRD §2）

### 2.1 外卖主流程（§2.1）
用户下单→用户支付→商户接单→商户出餐→骑手接单→骑手取餐→骑手配送→用户确认收货→订单完成→平台与商户/骑手分账结算

### 2.2 跑腿主流程（§2.2）
用户发起跑腿需求→系统价格预估→用户下单支付→系统派单/骑手抢单→骑手取件/完成帮买→骑手配送→收件人/用户确认收货→订单完成→平台与骑手分账结算

## 三、关键非功能约束
- 订单接口并发 ≥ 1000 TPS（PRD §4.1.2）
- 订单状态推送延迟 ≤ 3s（PRD §4.1.4）
- 多端数据实时同步无不一致（PRD §6.1.3）
- 全流程闭环、异常可追溯（PRD §6.1.2）

## 四、P4 任务范围

### 4.1 in-scope（8 大服务 + 闭环联调）
| 模块 | 交付 |
|---|---|
| order | 下单、取消、确认、状态机、库存扣减、定时关单、取送件码、售后联动 |
| product | 分类/商品/SKU/套餐/上下架/库存增减 |
| shop | 店铺 CRUD、营业时间、配送区域、自动接单、店铺公告 |
| dispatch | 智能派单（抢单+派单双模式）、顺路单匹配、转单、运力看板 |
| payment | 微信 JSAPI/APP 支付、支付宝、余额、退款、异步回调、对账 |
| finance | 分账规则、分账执行、账户流水、提现、发票、对账 |
| marketing | 优惠券模板、发放、核销、满减/折扣/新客/拼单、红包池、积分 |
| review | 评价、回复、申诉、投诉、售后工单、仲裁 |

### 4.2 out-of-scope
- 前端对接逻辑（P5~P8）
- 压测、灰度（P9）
- 报表 OLAP（P9+）

## 五、待澄清
| 编号 | 问题 | 默认 |
|---|---|---|
| Q4.1 | 派单算法首版 | 距离(40%) + 运力(30%) + 顺路(20%) + 评分(10%)；后续 A/B 迭代 |
| Q4.2 | 支付聚合收单 | 不接入聚合，直连微信 + 支付宝，保持低手续费 |
| Q4.3 | 分账时机 | T+1（订单 `finished_at + 1d` 触发），异常单需人工复核 |
| Q4.4 | 结算周期 | 商户 T+1；骑手周结或日结（管理后台可配） |
| Q4.5 | 跑腿定价 | 公式：起步价 + 距离阶梯 + 重量阶梯 + 夜间/恶劣天气系数，城市维度配置 |
| Q4.6 | 订单关单时间 | 未支付 15min、待接单 5min 商户未响应自动释放 |
| Q4.7 | 库存模式 | 默认 `无限库存 stock=-1`；商户可启用库存管理 |
