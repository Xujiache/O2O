# ALIGNMENT_P1_项目初始化

> 本文件用于 P1 阶段的**需求对齐**，回溯 PRD 原文、拆解、边界确认、列出待澄清问题。
> 严格基于《O2O外卖+跑腿一体化平台 需求规格说明书 V1.0》，不做任何偏离。

---

## 一、PRD 原文回溯

### 1.1 项目核心目标（PRD §1.2）

1. 实现外卖 + 跑腿双业务线的一体化流程管控，支持多端业务协同
2. 搭建高可用、高并发的即时配送系统，保障订单全流程的稳定性与实时性
3. 提供完善的运营、财务、风控管理能力，支撑平台规模化商业运营
4. 保障多端用户体验，满足用户便捷下单、商户高效管店、骑手灵活接单的核心诉求

### 1.2 核心角色（PRD §1.4）

- C 端用户（微信小程序）
- 入驻商户（APP / 小程序）
- 平台骑手（APP）
- 平台管理员（Web 管理后台）

### 1.3 技术栈规范（PRD §5，原文照录）

| 端 / 模块 | 技术栈规范 |
|---|---|
| 用户端（微信小程序）、商户端、骑手端 | uni-app + Vue3 + Pinia + uView Plus；用户端编译为微信小程序，商户/骑手端编译为 iOS+Android APP |
| 平台管理后台 | Vue3 + Vite + Element Plus + Pinia + Vue Router |
| 后端服务 | 主架构推荐 SpringCloud/SpringBoot 微服务架构，可选 NestJS（Node.js）；分模块服务化开发 |
| 数据库 | 主数据库 MySQL 8.0+；缓存数据库 Redis 6.0+；支持时序数据库存储骑手定位轨迹 |
| 中间件 | 消息队列 RabbitMQ/Kafka；对象存储 MinIO / 阿里云 OSS；分布式锁 Redisson |
| 第三方服务 | 高德/百度地图开放平台、微信开放平台/微信支付、支付宝开放平台、APP 推送服务（极光/个推）、短信服务 |
| 部署与运维 | Docker + Kubernetes 容器化部署；Nginx 反向代理；Jenkins CI/CD；Prometheus + Grafana 监控告警 |

### 1.4 非功能性需求摘要（PRD §4）

- 核心接口平均响应 ≤ 200ms，峰值 ≤ 500ms
- 订单并发 ≥ 1000 TPS
- 小程序首屏 ≤ 2s，页面切换 ≤ 300ms
- 骑手定位上报 ≥ 10s/次，订单状态推送延迟 ≤ 3s
- 系统可用性 ≥ 99.9%，故障恢复 ≤ 30min

## 二、P1 阶段任务拆解

### 2.1 范围定义

P1 阶段仅完成**项目脚手架与开发环境搭建**，不涉及具体业务功能：

1. 创建 5 端代码仓库结构（用户端/商户端/骑手端/管理后台/后端）
2. 落地各端脚手架（uni-app + NestJS 初始化）
3. 统一开发规范（ESLint、Prettier、Stylelint、Git Hooks、提交规范）
4. 配置开发环境变量（多环境 .env：dev / test / prod）
5. 搭建 CI/CD 基础骨架（Jenkinsfile / Dockerfile / docker-compose）
6. 完善 README 与 开发指南

### 2.2 输入

- PRD V1.0（已就位）
- 管理后台 art-design-pro 精简版（已就位于 `管理后台/`）

### 2.3 输出

- 5 端代码仓库初始结构
- 统一的 lint / commit / workflow 规范文件
- 基础 Dockerfile 与 docker-compose（可启动空骨架）
- 每端 README.md
- 项目总 README.md 或沿用 `说明文档.md`

### 2.4 约束

- 技术栈严格对齐 PRD §5
- 后端选型：NestJS（PRD 允许方案，效率最高）
- 管理后台**沿用已有**框架，不重建
- Node.js ≥ 20.19.0，pnpm ≥ 8.8.0（对齐管理后台 package.json）

## 三、边界与范围确认

### 3.1 P1 不做

- 任何业务代码（留到 P3~P8）
- 数据库建表（留到 P2）
- UI/UX 详细设计（留到各端开发阶段）
- 第三方服务对接（留到具体模块）

### 3.2 P1 必做

- 5 端代码目录结构创建完毕
- 每端可以 `pnpm install` + `pnpm dev` 启动空骨架
- 后端可以 `pnpm run start:dev` 启动并返回 hello
- Git 仓库初始化（或在单仓中清晰划分目录）

## 四、待澄清问题（需用户确认）

| 编号 | 问题 | 默认建议 |
|---|---|---|
| Q1 | 单仓（monorepo）还是多仓？ | 单仓：`O2O跑腿+外卖/` 下多端并列，使用 pnpm workspace 管理 |
| Q2 | 是否启用 pnpm workspace？ | 建议启用，便于共享类型定义 |
| Q3 | 后端微服务数量？ | P1 先建单 NestJS 应用，P3 根据基础服务拆分模块（monolith-first） |
| Q4 | 时序数据库选型？ | InfluxDB 或 TimescaleDB，P2 再定 |
| Q5 | 消息队列 RabbitMQ 还是 Kafka？ | RabbitMQ（订单中小规模异步，生态简单） |
| Q6 | 对象存储 MinIO 还是阿里云 OSS？ | 开发环境 MinIO，生产 OSS |
